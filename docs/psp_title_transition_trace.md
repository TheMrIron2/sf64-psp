# PSP Title Transition Frame Trace

This diagnostic trace investigates the PSP title-screen startup-to-steady-state
slowdown without changing rendering, timing, PSPGL, or title behaviour.

## Build

Frame trace is disabled by default:

```make
SF64_PSP_PROFILE_FRAME_TRACE ?= 0
SF64_PSP_PROFILE_FRAME_TRACE_FRAMES ?= 240
```

Frame trace requires `SF64_PSP_PROFILE_PHASES=1`; the Makefile errors out if
the trace is enabled without phase profiling. The frame capacity must be between
1 and 3600. The default 240-frame trace stores a fixed static buffer in
`src/psp/profiler.c`; there is no allocation or memory-stick I/O while frames
are being recorded.

Hardware trace build:

```bash
make USE_LOCAL_PSPGL=1 \
     SF64_PSP_PSPGL_DLIST_SIZE_WORDS=512 \
     SF64_PSP_PSPGL_PROFILE=0 \
     SF64_PSP_PROFILE_PHASES=1 \
     SF64_PSP_PROFILE_FRAME_TRACE=1 \
     SF64_PSP_PROFILE_FRAME_TRACE_FRAMES=240 \
     SF64_PSP_PROFILE_CAPTURE_FRAMES=240 \
     PSP_FPS_OVERLAY=0 \
     BUILD_DIR=build/psp-title-transition-trace \
     psp
```

Control build:

```bash
make USE_LOCAL_PSPGL=1 \
     SF64_PSP_PSPGL_DLIST_SIZE_WORDS=512 \
     SF64_PSP_PSPGL_PROFILE=0 \
     SF64_PSP_PROFILE_PHASES=1 \
     SF64_PSP_PROFILE_FRAME_TRACE=0 \
     PSP_FPS_OVERLAY=0 \
     BUILD_DIR=build/psp-title-trace-off \
     psp
```

## Output

The existing aggregate files remain:

```text
profile-NNN.csv
profile-NNN.txt
```

When frame trace is enabled, capture stop also writes:

```text
profile-NNN-frames.csv
profile-NNN-phases.csv
profile-NNN-frame-categories.csv
```

`profile-NNN-frames.csv` has one row per graphics-task frame boundary. It
includes raw and timer-adjusted whole-frame phase totals, global frame context,
title markers, summary phase mappings, workload deltas, and
`incomplete_phase_mask`/`incomplete_phase_depth`.

`profile-NNN-phases.csv` is long-form:

```text
frame,phase,calls,total_us_raw,total_us_adjusted,items
```

`profile-NNN-frame-categories.csv` is long-form:

```text
frame,category,name,count
```

It contains per-frame deltas for batch flush reasons, batch-state transitions,
and texture-flush sources.

## Phase Summary Mapping

The summary CSV derives columns from existing phases:

```text
graphics_task_us                    -> graphics task total
display_list_processing_us          -> display-list traversal and dispatch
batch_flush_us                      -> batch flush total
vertex_processing_or_upload_us      -> G_VTX processing total + PSPGL vertex stream upload
lighting_us                         -> G_VTX processing total
clipping_us                         -> software clipping/subdivision
texture_processing_or_upload_us     -> texture lookup/decode/preparation + texture decode/conversion + texture upload
pspgl_state_setup_us                -> PSPGL state/setup
draw_submission_us                  -> PSPGL draw submission
gl_flush_us                         -> glFlush queue flush
finish_or_sync_us                   -> graphics finish/synchronisation
```

There is no separate existing lighting-only phase. `lighting_us` is therefore
the existing `G_VTX processing total` phase, while the workload counters expose
`lit_vertices` and `lighting_evaluations` for per-frame correlation.

after each established graphics-task completion boundary and do not reset the
aggregate totals.

Texture cache key tracking remains aggregate-only; per-frame trace records the

## Title Source Findings

The title implementation is in `src/overlays/ovl_menu/fox_title.c`.

Top-level state:

```text
gTitleState
TITLE_IDLE=0
TITLE_SETUP=1
TITLE_UPDATE_ENTRY=2
```

`Title_Main()` advances the top-level title state. `Title_Setup()` sets
`gGameFrameCount = 0`, `gTitleState = TITLE_UPDATE_ENTRY`, and, on PSP, forces
`gGoToTitle = true`, selecting `sCutsceneState = TITLE_SCREEN` instead of the
opening cinematic path.

Title-scene state:

```text
sCutsceneState
TITLE_SCREEN=0
TITLE_GREAT_FOX_TRAVELING=1
TITLE_CS_TEAM_RUNNING=2
TITLE_GREAT_FOX_CLOSE_UP=3
TITLE_TAKE_OFF=4
TITLE_TAKE_OFF_SPACE=5
TITLE_RANKING=7
```

Local title-screen state is `sSceneState` inside `Title_Screen_Update()`.
For the static four-character title screen:

```text
0 -> calls Title_Screen_Setup(), initializes title state and lighting/camera
1 -> eases camera/light angles until D_menu_801B86CC reaches 57.0f
2 -> waits for D_menu_801B82B0 to count down from 100 to 0
3 -> randomly selects a character for idle motion and starts a timer
4 -> waits for that timer, smooths the selected character back, then returns to 3
```

The left-to-right lighting and face-look markers are raw variables, not renamed
behavioural conclusions:

```text
D_menu_801B86C8 / D_menu_801B86CC  light rotation inputs to Title_SetLightRot()
D_menu_801B82E0 / D_menu_801B82E4 / D_menu_801B82E8  computed light direction
D_menu_801B905C / D_menu_801B9060 / D_menu_801B9064  face/look target used by Title_Team_Draw()
D_menu_801B8304 / D_menu_801B8308 / D_menu_801B830C  title ambient RGB setup values
```

`Title_Screen_Draw()` sets `gLight1R/G/B` and `gAmbientR/G/B`, optionally eases
`D_menu_801B86C8/D_menu_801B86CC` toward the face/look target when
`D_menu_801B9040 != 0`, calls `Title_SetLightRot()`, then draws the four team
members every frame in this order: Fox, Falco, Slippy, Peppy. The Arwing is
drawn afterward with a second light setup.

Character models are not progressively introduced on the PSP title-screen path:
`Title_Screen_Draw()` calls `Title_Team_Draw()` for all four characters every
frame once `TITLE_SCREEN` is active. Character animation/motion changes are
controlled by `sTitleTeam[i].frameCount`, `sTitleTeam[i].unk_5C`, and
`sTitleTeam[i].unk_54`; scene states 3 and 4 select and animate one character
at a time after the initial hold timer reaches zero.

Audio work starts during title setup through:

```text
AUDIO_SET_SPEC(SFX_LAYOUT_DEFAULT, AUDIOSPEC_TITLE)
AUDIO_PLAY_BGM(NA_BGM_TITLE)
```

No title-rendering optimization or timing change is encoded by the trace. The
trace records raw title state, timers, light values, camera values, and workload
deltas so hardware captures can align the slowdown with actual state changes.

## Hardware Procedure

1. Launch the game.
2. Trigger capture at the earliest available opportunity, before the title scene
   begins.
3. Let the full 240-frame capture span pre-title transition, initial title,
   visible slowdown, and warm steady-state title.
4. Do not press buttons or leave the title screen during capture.
5. Repeat three times for alignment and reproducibility.

Frame trace is diagnostic. Compare frame-to-frame within the same trace build;
do not use it as an absolute performance comparison against non-trace builds.
